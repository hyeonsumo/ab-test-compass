## 1. 프로젝트 배경

### 1.1. 만든 이유

Statsig, Hackle, Eppo 같은 A/B 테스트 도구를 사용해보면 결과 화면에  
"p-value 0.012", "신뢰구간 [+1.1%, +5.3%]", "SRM 정상" 같은 숫자들이 표시됩니다.

그러나 다음과 같은 질문들에 명확히 답하기 어려웠습니다.

- 같은 사용자가 새로고침해도 왜 같은 그룹으로 배정되는가
- p-value는 어떻게 계산되며, 왜 0.05가 기준인가
- 신뢰구간과 p-value는 같은 표준오차를 사용하는가
- 50:50으로 설정했는데 53:47로 나오면 그게 문제인지 어떻게 판단하는가

문서를 읽어도 추상적으로만 이해되어, 직접 구현해보며 이해하기 위해 시작한 프로젝트입니다.

### 1.2. 목표

상용 도구를 따라잡는 것이 아니라, 그것들이 추상화한 핵심 메커니즘을  
코드 레벨에서 직접 짜보며 이해하는 것을 목표로 했습니다.

## 2. 학습 노트

만들면서 알게 된 것들을 정리한 섹션입니다.

### 2.1. 결정적 해싱(Deterministic Hashing)의 동작 원리

처음에는 "랜덤으로 A/B를 나누면 되지 않나"라고 생각했으나, 그러면 같은 사용자가  
재방문할 때 다른 그룹으로 배정되어 실험이 오염된다는 점을 알게 되었습니다.

```python
def assign_variant(user_id, salt, ratio):
    key = f"{user_id}:{salt}"
    bucket = int(hashlib.md5(key.encode()).hexdigest(), 16) % 10000 / 10000
    return "A" if bucket < ratio else "B"
```

`user_id + salt`를 해싱한 결과를 0~1 사이로 정규화해서 비율로 자르는 방식.  
**같은 입력은 항상 같은 출력**이라는 해시 함수의 성질을 이용한 것입니다.  
서버 호출 없이 클라이언트가 즉시 결정할 수 있고, salt를 실험별로 다르게 두면  
실험 간 배정이 독립적이 됩니다.

### 2.2. 풀링된 표준오차와 비풀링 표준오차의 구분

z-검정과 신뢰구간 모두 표준오차를 사용하지만, **서로 다른 표준오차를 사용**해야 한다는  
것을 직접 구현하며 알게 되었습니다.

- **p-value 계산 (z-검정)**: 귀무가설은 "두 그룹의 전환율이 같다". 같다고 가정해야 하므로  
  두 그룹을 합쳐 공통 비율로 추정 → **풀링된 표준오차**
- **신뢰구간 계산**: "두 그룹이 다르다"고 가정하고 차이의 범위를 추정 → **비풀링 표준오차**

같은 데이터를 두 가지 가정으로 분석하는 것이며, 통계 분석에서 자주 놓치는 디테일입니다.

### 2.3. SRM(Sample Ratio Mismatch) 검사의 의미

50:50으로 설정했는데 실제로 51:49가 나오면 정상인지, 비정상인지를  
카이제곱 검정으로 판단합니다.

Microsoft, Booking.com의 실험 가이드에서 임계값으로 **p < 0.001**을 사용한다는  
것을 알게 되었고, 이것이 단순한 "약간의 차이"가 아니라 **배정 로직 버그나 데이터  
수집 누락**을 의심해야 하는 강한 신호라는 것을 이해했습니다.

상용 도구의 "Data Quality" 알림이 사실 이 검정이었다는 점도 직접 구현하며 알았습니다.

### 2.4. 엿보기 편향(Peeking)의 위험성

처음에는 "p-value가 0.04로 떨어지면 유의한 것 아닌가"라고 생각했습니다.

직접 시뮬레이션을 돌려보니, **효과가 전혀 없는 두 그룹**으로 실험을 돌려도 표본이  
작을 때는 p-value가 우연히 0.05 아래로 떨어지는 순간이 자주 발생했습니다.  
그 순간을 잡아 종료하면 잘못된 결론에 도달합니다.

이 직관을 사용자에게도 보여주기 위해, 결과 화면에서 표본이 권장치에 미달하면  
명시적으로 경고하도록 구현했습니다.

### 2.5. 한국 실무 문헌에서 얻은 교훈

만들면서 한국 IT 기업들의 기술 블로그를 다수 참고했습니다.

- **카카오페이**: 자체 A/B 도구를 만들 때 MVP부터 시작해 점진 확장한 흐름.  
  이 원칙이 본인 프로젝트의 범위 설정에 영향을 주었습니다.
- **토스**: "문제 정의 → 가설 → EDA → 실험 → 확장" 흐름.  
  가설을 명시적으로 적는 것이 실험의 출발점이라는 점이, AI 어시스턴트로 가설을  
  정형화하는 기능을 만든 이유가 되었습니다.
- **Hackle**: "p-value 0.06인데 종료해야 할까?" 같은 실무 고민이 문서화되어 있었고,  
  엿보기 편향 방지 기능을 넣은 동기가 되었습니다.

## 3. 주요 기능

### 3.1. 핵심 메커니즘

- **결정적 해싱 배정**: MD5 기반, 실험별 salt로 독립성 보장
- **가상 트래픽 시뮬레이션**: baseline 및 treatment effect 기반 확률적 전환 생성
- **통계 분석**: 풀링된 z-검정, 비풀링 신뢰구간, p-value
- **SRM 감지**: 카이제곱 검정 (p < 0.001 임계)
- **엿보기 편향 방지**: 필요 표본 미달 시 결과 화면에서 경고

### 3.2. AI 가설 설계 어시스턴트

Google Gemini를 활용해 자연어 가설을 정형화된 실험 설계로 변환합니다.

- 자연어 입력을 받아 가설 문장, 변형 설명, 측정 지표, baseline, 권장 기간 자동 생성
- 통제 변수 위반(여러 요소 동시 변경) 등 실무 함정을 사전 점검
- 테마별(결제/UI/메시지 등) 컨텍스트 주입으로 도메인에 맞는 추천 제공

### 3.3. 시각화

- 그룹별 비교 막대 차트 (전환율 비교)
- 누적 전환율 추이 차트 (표본 증가에 따른 수렴 양상)

## 5. 프로젝트 범위

### 5.1. 포함된 범위

본 프로젝트는 A/B 테스트의 **핵심 메커니즘 이해**에 집중했습니다.

- 사용자 그룹 배정 알고리즘
- 노출/전환 이벤트 수집 및 저장
- 통계적 유의성 검정 (z-검정, 신뢰구간)
- 데이터 품질 검증 (SRM)
- 결과 시각화 및 해석

### 5.2. 다루지 않은 범위

다음 항목들은 본 학습 단계에서 다루지 않았습니다.  
핵심 메커니즘을 먼저 확실히 이해한 뒤, 후속 단계에서 학습할 주제입니다.

- 멀티 실험 동시 운영, 멀티테넌시
- 피처 플래그 및 점진 롤아웃
- 외부 SDK / HTTP API 연동
- CUPED, 순차 검정, 베이지안 분석
- 사용자 인증 및 권한 관리

## 7. 시작하기

### 7.1. 사전 요구사항

- Python 3.10 이상
- (선택) Google Gemini API 키 — AI 어시스턴트 사용 시 필요

### 7.2. 설치 및 실행

```bash
git clone https://github.com/hyeonsumo/ab-test-compass.git
cd ab-test-compass

python -m venv venv
venv\Scripts\activate          # Windows
# source venv/bin/activate     # Mac/Linux

pip install -r requirements.txt
python app.py
```

### 7.3. 실행 파일 (Windows)

소스 빌드 없이 바로 실행하려면 `dist/ABTestSandbox.exe`를 더블클릭합니다.

### 7.4. AI 어시스턴트 활성화 (선택)

1. https://aistudio.google.com/apikey 에서 Gemini API 키 발급
2. 앱 첫 실행 시 환영 화면에서 키 입력  
   (또는 사이드바의 `AI 설정`에서 변경 가능)

키는 로컬 PC의 `data/config.json`에만 저장되며 외부로 전송되지 않습니다.

## 8. 향후 학습 계획

만들면서 떠오른 미해결 질문들. 다음 단계로 학습할 주제들입니다.

- **CUPED**: 사전 데이터로 분산을 줄여 실험 기간을 단축하는 기법
- **순차 검정(Sequential Testing)**: 엿보기 편향 없이 조기 종료를 허용하는 통계 방법
- **베이지안 A/B 검정**: "B가 더 나을 확률 X%"와 같은 직관적 해석 방식
- **멀티 암드 밴딧(MAB)**: 실험 중에도 우세한 안에 트래픽을 더 보내는 적응적 알고리즘
