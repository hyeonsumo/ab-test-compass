## 1. 프로젝트 개요

| 항목 | 내용 |
|---|---|
| 프로젝트명 | AB Compass |
| 개발 기간 | 2026.03 ~ 2026.05 |
| 구성 | 개인 프로젝트 · 1인 개발 |

Statsig, Hackle, Eppo 같은 상용 A/B 테스트 플랫폼의 핵심 메커니즘

1인 학습 도구 규모로 재구성. Gemini API를 활용한 가설 설계 어시스턴트 포함.

## 2. 프로젝트 시작 배경

업무에서 A/B 테스트 도구(Statsig, Hackle)를 다루면서 결과 화면의 통계량을  
표면적으로만 해석하게 되는 한계를 느꼈다. 핵심 메커니즘을 직접 구현해 보는 것이  
가장 빠른 학습 경로라 판단하여 이 프로젝트를 시작했다.

**해소되지 않았던 네 가지 질문**

- 같은 사용자가 재방문해도 동일 그룹에 배정되는 메커니즘
- p-value의 계산 절차와 0.05 임계값의 근거
- 신뢰구간과 p-value가 같은 표준오차를 쓰는지 여부
- 50:50 설정에서 53:47 분포가 나왔을 때의 정상성 판단 기준

상용 도구 문서로는 추상적인 수준에서만 이해되어, 핵심 메커니즘을  
직접 구현해 검증하는 방식으로 진행.

## 3. 주요 분석

A/B 테스트 플랫폼의 동작 원리를 4개 구성요소로 분해해 구현.  
각 구성요소는 상용 도구가 추상화한 핵심 로직을 직접 코드로 재현한 결과.

### 3-1. 사용자 그룹 배정 알고리즘

랜덤 배정은 동일 사용자가 재방문할 때 그룹이 바뀌어 실험이 오염된다.  
해시 함수의 결정성을 활용해 일관된 배정을 구현.

```python
def assign_variant(user_id, salt, ratio):
    key = f"{user_id}:{salt}"
    bucket = int(hashlib.md5(key.encode()).hexdigest(), 16) % 10000 / 10000
    return 'A' if bucket < ratio else 'B'
```

- **일관성**: 동일 사용자는 언제 접속해도 같은 그룹
- **독립성**: 실험별 salt 분리로 실험 간 배정 상관관계 제거
- **무상태성**: 서버 조회 없이 클라이언트에서 즉시 결정

Statsig, LaunchDarkly 같은 상용 플랫폼의 표준 방식과 동일.

### 3-2. 유의성 검정 및 효과 추정

두 그룹의 전환율 차이가 우연인지 실제 효과인지 판정하기 위해 z-검정으로  
p-value를 계산하고, 효과 크기와 불확실성 범위를 95% 신뢰구간으로 추정.

| 산출 지표 | 활용 |
|---|---|
| p-value | 차이가 우연일 확률 (유의성 판정) |
| 신뢰구간 | 효과 크기의 추정 범위 |
| z-score | 검정 통계량 |
| Lift | A 대비 B의 상대적 개선율 |

검정과 추정은 서로 다른 통계 가정을 쓰기 때문에 표준오차를 분리해 계산.  
같은 데이터에 두 가지 분석을 적용해야 하는 디테일까지 반영.

### 3-3. 데이터 품질 자동 진단

실험 설정 비율과 실제 배정 비율이 일치하는지 카이제곱 검정으로 검증.  
50:50 설정에서 53:47이 나왔을 때 우연인지 배정 로직의 결함인지 통계적으로 판단.

- 임계값: `p < 0.001` (Microsoft, Booking.com 실험 가이드 관례)
- 임계값 초과 시 분석 결과 신뢰 전 데이터 파이프라인 점검 필요
- 상용 도구의 "Data Quality 알림"이 사실상 같은 검정의 결과

### 3-4. 의사결정 함정 차단

표본이 권장치에 미달한 상태에서 p-value를 조기 확인하고 종료하면  
false positive 비율이 명목 유의수준(5%)을 크게 넘어선다.

**시뮬레이션 검증**

효과가 같은 두 그룹으로 실험을 반복할 때, 표본 누적 과정에서 p-value가  
일시적으로 0.05 아래로 떨어지는 순간이 자주 발생. 이 시점을 종료 기준으로  
잡으면 false positive 비율이 약 20~30%까지 올라가는 것을 시뮬레이션으로 재현.

권장 표본 미달 시 결과 화면에 경고를 노출해 잘못된 의사결정을 사전 차단.

### 3-5. 참고 문헌

국내 IT 기업의 실험 운영 사례와 통계 방법론 표준 문헌을 참조.

- **카카오페이 기술블로그** — 자체 A/B 테스트 플랫폼 구축 시 MVP 접근법
- **토스 기술블로그** — "문제 정의 → 가설 → EDA → 실험 → 확장" 운영 흐름
- **Hackle 기술블로그** — 실무 의사결정에서의 통계 함정 사례
- **Kohavi et al., Trustworthy Online Controlled Experiments (2020)** — SRM 임계값 근거

## 4. 주요 기능

### 4-1. 핵심 기능

- **사용자 배정** — MD5 기반 결정적 해싱, salt로 실험 간 독립성 확보
- **트래픽 시뮬레이션** — baseline과 treatment effect 기반 확률적 전환 생성
- **유의성 검정** — z-검정, 95% 신뢰구간, p-value 산출
- **데이터 품질 점검** — 카이제곱 검정으로 SRM 자동 감지
- **표본 충분성 경고** — 권장 표본 미달 시 의사결정 함정 경고
- **결과 시각화** — 그룹별 비교 막대 차트, 누적 전환율 추이 차트

### 4-2. AI 가설 설계 어시스턴트

Gemini API로 자연어 가설을 정형화된 실험 설계로 변환.  
통제 변수 원칙, 측정 가능성, 실무 함정을 시스템 프롬프트에 포함시켜  
변환과 동시에 사전 검증을 수행하도록 설계.

```text
입력  : "결제 버튼 색상을 바꾸면 전환율이 오를 것 같다"
↓
출력  : 정형 가설 · A/B 변형 정의 · 측정 지표 · baseline 추천 ·
        권장 기간 · 통제 변수 위반 여부
```

## 6. 분석 범위 및 한계

### 6-1. 분석 범위

A/B 테스트의 **핵심 메커니즘 검증**에 집중.

- 사용자 그룹 배정 알고리즘
- 노출 및 전환 이벤트 수집 · 저장
- 통계적 유의성 검정 (z-검정, 신뢰구간)
- 데이터 품질 검증 (SRM)
- 결과 시각화 및 해석

### 6-2. 한계점

아래 항목은 이번 단계에서 다루지 않음. 핵심 메커니즘 검증을 우선 목표로  
잡았기 때문에, 분산 환경과 고급 통계 기법은 후속 과제로 분리.

- 멀티 실험 동시 운영 및 멀티테넌시
- 피처 플래그 및 점진적 롤아웃
- SDK 및 HTTP API 외부 연동
- CUPED, 순차 검정, 베이지안 분석 등 고급 통계 기법
- 사용자 인증 및 권한 관리

## 7. 실행 방법

### 7-1. 사전 요구사항

- Python 3.10 이상
- (선택) Google Gemini API 키 — AI 어시스턴트 사용 시 필요

### 7-2. 소스 빌드

```bash
git clone https://github.com/hyeonsumo/ab-test-compass.git
cd ab-test-compass

python -m venv venv
venv\Scripts\activate          # Windows
# source venv/bin/activate     # Mac/Linux

pip install -r requirements.txt
python app.py
```

### 7-3. 실행 파일 (Windows)

`dist/ABCompass.exe` 더블클릭으로 실행.

### 7-4. AI 어시스턴트 활성화 (선택)

1. https://aistudio.google.com/apikey 에서 Gemini API 키 발급
2. 앱 첫 실행 시 환영 화면에서 키 입력  
   (또는 사이드바의 'AI 설정'에서 변경 가능)

키는 로컬 `data/config.json`에 저장되며, Gemini API 호출 시 Google로 전달됨.

## 8. 향후 확장 계획

후속 학습 단계에서 다룰 주제.

- **CUPED** — 사전 데이터를 활용한 분산 감소 기법
- **순차 검정** — 엿보기 편향 없이 조기 종료를 허용하는 통계 방법 (mSPRT, always-valid p-value)
- **베이지안 A/B 분석** — "B가 우세할 확률 X%" 형태의 확률 기반 해석
- **멀티 암드 밴딧 (MAB)** — 실험 중 적응적 트래픽 배분 (Thompson Sampling)
